# Investment Advisor CRM

A full-stack broker productivity tool built on Next.js 15, Vercel Postgres, and Gemini 2.5 Flash. Financial advisors can manage their client book, drill into individual client holdings across investment wrappers, and query the database in plain English.

## Features

### Advisor Dashboard (`/dashboard`)
- **Advisor selector** — switch between advisors; all KPIs, charts, and the client table update to the selected book via live SQL
- **AI morning briefing** — six expandable cards (Investment Performance, Client Book, Economy & Markets, Client Activity, Advisor Priorities, Risk Overview), each with a data-specific headline and a full paragraph generated by Gemini and cached per advisor
- **KPI cards** — My AUM, Clients, Active Policies, Avg 1Y Return, Monthly Revenue, At-Risk Count; all advisor-scoped
- **Alert strip** — Risk Mismatches, Revenue at Risk, and estimated Potential Commission; clicking a card re-sorts the client table
- **Client Intelligence Table** — sortable by AUM, Potential Annual Commission, or Risk Priority; filterable by All / Pre-Retirement / Post-Retirement; the dynamic column swaps to Drawdown Rate (post-retirement) or Years to Retire (pre-retirement)

### Client Detail (`/clients/[clientId]`)
- **Profile, portfolio snapshot, alerts panel** — wrapper-aware alerts (LA drawdown, missing beneficiary, RA contribution gap, TFSA not funded, etc.) merged with policy-level alerts
- **Commission breakdown** — estimated annual commission totals plus a per-policy breakdown table driven by the shared commission engine
- **Investment Wrappers** — each legal/tax container shown as its own card (RA, TFSA, Living Annuity, Endowment, Preservation Fund, Unit Trust, Guaranteed Annuity) with a fund holdings table inside each
- **Living Annuity sustainability** — drawdown rate colour-coded (≤5% green · 5–7.5% amber · >7.5% red) with projected years to capital depletion at an assumed 6% return
- **Recent transactions** table
- **Communication workspace** — draft and manage emails and meeting requests per client

### Natural Language Query (`/query`)
- Type plain English; the app generates a Postgres SELECT, runs it, and renders a Recharts chart with an AI-generated explanation
- The LLM system prompt includes the full 16-table schema with SA-specific domain rules

## Getting Started

### 1. Install dependencies

```bash
pnpm install
```

### 2. Configure environment variables

```bash
cp .env.example .env
```

Fill in `.env`:

```
GOOGLE_GENERATIVE_AI_API_KEY=   # Gemini API key
GEMINI_MODEL=gemini-2.5-flash
POSTGRES_URL=
POSTGRES_PRISMA_URL=
POSTGRES_URL_NO_SSL=
POSTGRES_URL_NON_POOLING=
POSTGRES_USER=
POSTGRES_HOST=
POSTGRES_PASSWORD=
POSTGRES_DATABASE=
```

### 3. Seed the database

```bash
pnpm run seed
```

Drops and recreates all 16 tables, then populates them with 5 advisors, 50 clients, 30 funds, and realistic wrapper/holding/transaction data. Re-run any time to reset to a clean state.

### 4. Generate the AI briefing cache

The morning briefing and fund analytics are cached in the `dashboard_insights` table. Populate it after seeding:

```bash
curl -X POST http://localhost:3000/api/refresh-insights
```

Or click the **Refresh** button on the dashboard once the dev server is running.

### 5. Start the dev server

```bash
pnpm dev
```

Open [http://localhost:3000](http://localhost:3000).

## Routes

| Route | Description |
|---|---|
| `/dashboard?advisor=N` | Advisor cockpit — KPIs, morning briefing, client intelligence table |
| `/clients?advisor=N` | Full client list for an advisor |
| `/clients/[clientId]?advisor=N` | Client detail — wrappers, holdings, alerts, communication drafts |
| `/query` | Natural language SQL query interface |
| `POST /api/refresh-insights` | Regenerates the AI briefing and fund analytics cache |

## Database Schema

16 tables across three domains:

**Fund data** — `sector`, `peer_group`, `fund`, `period_definition`, `fund_performance_fact`, `fund_risk_fact`, `fund_flow_fact`, `fund_ranking_fact`, `peer_group_stat_fact`

**CRM data** — `advisor`, `client`, `policy`, `transaction`, `advisor_aum`

**Wrapper/holding data** — `wrapper` (investment product containers per client), `fund_holding` (individual fund positions within each wrapper)

Key conventions:
- Returns and fees are stored as decimals (`0.12` = 12%)
- All monetary amounts are in South African Rands (ZAR)
- `peer_group_quartile`: 1 = top, 4 = bottom
- `wrapper.drawdown_rate_pct`: fraction (`0.075` = 7.5%)
- `fund_holding.allocation_pct`: fraction (`0.45` = 45%)
- `wrapper.phase`: `'accumulation'` or `'drawdown'`
- `wrapper.wrapper_type`: `'retirement_annuity'`, `'tfsa'`, `'endowment'`, `'living_annuity'`, `'preservation_fund'`, `'unit_trust'`, `'guaranteed_annuity'`

## Architecture Notes

Two data paths are kept strictly separate:

| Path | Source | Cached | Advisor-scoped |
|---|---|---|---|
| KPIs, client table, wrapper holdings | `lib/advisor-data.ts` (live SQL, no LLM) | No | Yes |
| Fund analytics + AI morning briefing | `lib/insights.ts` + `dashboard_insights` table | Yes (manual refresh) | Per advisor |

The dashboard fetches both in parallel. Advisor-scoped queries are fast; fund analytics and briefings come from the cache.
