Context

 The current dashboard shows firm-wide aggregate data and is not useful to an individual advisor. The goal is to transform it into a broker
 productivity tool: one advisor's book at a time, with a prominent client intelligence table that ranks clients by priority so the advisor knows
 exactly who to call and why.

 Three concrete problems to fix alongside the redesign:
 1. Narrative is empty — generateCaption uses maxOutputTokens: 120, which is far too short for 3–4 sentences. The text gets truncated mid-sentence.
 2. KPI counts are wrong — page.tsx multiplies total_clients * 5 and monthly_revenue * 5 as a hack; this goes away with proper advisor scoping.
 3. No advisor filter — every SQL query in insights.ts is firm-wide.

 Architecture

 Two data paths — keep them separate:

 ┌────────────────────────────────────────────────────────────┬──────────────────────────────────────────┬───────────┬────────────────────────────┐
 │                            Path                            │                  Source                  │   Cache   │       Advisor-scoped       │
 ├────────────────────────────────────────────────────────────┼──────────────────────────────────────────┼───────────┼────────────────────────────┤
 │ Advisor KPIs + client table + book charts                  │ lib/advisor-data.ts (fresh SQL, no LLM)  │ No        │ Yes — filtered by          │
 │                                                            │                                          │           │ advisor_id                 │
 ├────────────────────────────────────────────────────────────┼──────────────────────────────────────────┼───────────┼────────────────────────────┤
 │ Fund analytics (top funds, Sharpe, flows, quartiles,       │ lib/insights.ts + dashboard_insights     │ Yes       │ No — firm-wide             │
 │ narrative)                                                 │ table                                    │ (cron)    │                            │
 └────────────────────────────────────────────────────────────┴──────────────────────────────────────────┴───────────┴────────────────────────────┘

 The dashboard page fetches both in parallel on every load. The advisor-scoped queries are fast (no LLM); fund analytics come from cache.

 Files to Create

 lib/advisor-data.ts (new)

 All advisor-scoped SQL. No LLM, no cache. Four exports:

 export interface AdvisorInfo { advisor_id: number; advisor_name: string; branch: string; region: string }
 export interface AdvisorKpis { my_aum; client_count; active_policy_count; avg_1y_return_pct; monthly_revenue; at_risk_count }
 export interface ClientRow {
   client_id; client_name; risk_profile; status; client_since;
   total_aum; policy_count;
   avg_1y_return_pct;   // weighted by policy value
   avg_quartile;         // 1=best, 4=worst
   last_activity;        // MAX(transaction_date)
   has_risk_mismatch;    // conservative in SA Equity OR aggressive in Money Market
   potential_annual_commission; // estimated from current policy value x policy-type annual commission rate
 }
 export interface AdvisorBookStats {
   risk_profile_breakdown: { risk_profile, count }[];
   policy_type_distribution: { policy_type, count }[];
   transaction_activity: { month, tx_count, total_amount }[];
 }

 export async function getAdvisors(): Promise<AdvisorInfo[]>
 export async function getAdvisorKpis(advisorId: number): Promise<AdvisorKpis>
 export async function getAdvisorClients(advisorId: number): Promise<ClientRow[]>
 export async function getAdvisorBookStats(advisorId: number): Promise<AdvisorBookStats>

 Key SQL for getAdvisorClients (single query with CTEs):
 SELECT
   c.client_id,
   c.first_name || ' ' || c.last_name AS client_name,
   c.risk_profile, c.status, c.client_since,
   SUM(p.current_value)::NUMERIC AS total_aum,
   COUNT(DISTINCT p.policy_id)::INT AS policy_count,
   ROUND((SUM(fpf.return_annualized * p.current_value)
     / NULLIF(SUM(p.current_value),0) * 100)::NUMERIC, 1) AS avg_1y_return_pct,
   ROUND(AVG(frf.peer_group_quartile)::NUMERIC, 1) AS avg_quartile,
   MAX(t.transaction_date) AS last_activity,
   BOOL_OR(
     (c.risk_profile = 'conservative' AND f.sector_id = 1) OR
     (c.risk_profile = 'aggressive'   AND f.sector_id = 4)
   ) AS has_risk_mismatch
 FROM client c
 JOIN policy p ON p.client_id = c.client_id
 JOIN fund f   ON f.fund_id = p.fund_id
 LEFT JOIN fund_performance_fact fpf ON fpf.fund_id = f.fund_id
   AND fpf.period_id = (SELECT period_id FROM period_definition WHERE period_code = '1Y')
 LEFT JOIN fund_ranking_fact frf ON frf.fund_id = f.fund_id
   AND frf.period_id = (SELECT period_id FROM period_definition WHERE period_code = '1Y')
 LEFT JOIN transaction t ON t.policy_id = p.policy_id
 WHERE c.advisor_id = $1
 GROUP BY c.client_id, c.first_name, c.last_name, c.risk_profile, c.status, c.client_since
 ORDER BY total_aum DESC

 Risk mismatch logic:
 - conservative client holding any SA Equity fund (sector_id = 1) → mismatch
 - aggressive client holding any Money Market fund (sector_id = 4) → mismatch

 Potential annual commission is calculated by the shared commission engine,
 using current policy value multiplied by the configured annual rate for each policy type.

 components/dashboard/ClientIntelligenceTable.tsx (new, client component)

 Sortable table. Sort state is local React state (no URL, no server round-trip).

 Sort modes:
 - "aum" — ORDER BY total_aum DESC (default)
 - "commission" — ORDER BY potential_annual_commission DESC
 - "risk" — risk_mismatch first, then dormant/inactive, then AUM DESC

 Column rendering:
 - AUM → formatZar()
 - 1Y Return → coloured: green ≥ 10%, amber 5–10%, red < 5%
 - Avg Quartile → coloured: Q1 green, Q2 blue, Q3 amber, Q4 red; show "1.0★" etc
 - Risk Alignment → green "Aligned" badge OR red "Mismatch" badge
 - Last Activity → relative date ("3 mo ago") or "No activity"
 - Status → coloured badge: active=green, dormant=amber, inactive=red
 - ⚑ flag → shows if any of: risk_mismatch, status≠active, avg_quartile > 3
